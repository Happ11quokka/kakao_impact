# EXIF 기반 사진 이벤트 그룹화 + 이벤트별 분석 설계

## 배경 및 목적

사용자가 카카오 챗봇에 사진을 여러 장 보낼 때, 현재는 모든 사진이 하나의 묶음으로 처리된다. 그러나 실제로는 사진들이 서로 다른 시간/장소(이벤트)에서 촬영되었을 수 있다. 예: 점심 카페에서 3장 + 저녁 식당에서 2장.

이 기능은 사진의 EXIF 메타데이터에서 촬영 시간을 추출해 10분 간격으로 이벤트 단위로 자동 그룹화하고, 이벤트별로 분석/저장을 진행한다. 감정분류 모드와 단순기록 모드 모두에 적용된다.

## 적용 범위

- 감정분류 모드 + 사진 2장 이상 + 이벤트 2개 이상으로 그룹화
- 단순기록 모드 + 사진 2장 이상 + 이벤트 2개 이상으로 그룹화

사진 1장이거나 모든 사진이 같은 이벤트로 묶이면 **기존 흐름을 그대로 사용**한다.

## 동작 흐름

### 감정분류 모드

1. 사용자가 사진을 N장 전송 → `pending_photo.urls`에 누적 (Plan A 동작 유지)
2. 사용자가 텍스트 입력 → 챗봇이 모든 사진을 다운로드하고 EXIF 시간 추출
3. 10분 간격으로 그룹화 → 그룹이 1개면 기존 감정분류 흐름 진입, 2개 이상이면 순차 질문 모드 진입
4. 순차 질문 모드:
   - 챗봇: "오늘 N개의 이벤트로 나뉘었어요. 이벤트 1 (11:00 · 3장): 이때 어떤 느낌이었나요?"
   - 사용자: 답변 입력
   - 챗봇: `classify_emotion_with_supervisor`로 분류 → 응답에 "맞아요/다시 찾을게요" 버튼
   - 사용자: "맞아요" → 해당 이벤트 원석 저장 → 다음 이벤트 질문 진행
   - 마지막 이벤트 저장 완료 시 "오늘 N개의 이벤트가 모두 저장됐어요! ✨" 응답

### 단순기록 모드

1. 사용자가 사진을 N장 전송 → `pending_simple_photo_buffer`에 누적 (기존 즉시 저장 동작 변경)
   - 첫 사진: "사진 1장 모았어요! 텍스트나 다른 사진을 더 보내주세요."
   - 추가 사진: "사진 N장 모였어요!"
2. 사용자가 텍스트 입력 → EXIF 파싱 → 10분 간격으로 그룹화
3. 각 이벤트를 단순기록으로 저장:
   - 주 사진 + 사용자 텍스트 + 이벤트 시간 → 1개의 `chatbot` 레코드 (gem='단순기록', record_text=텍스트, image_url=주사진, has_photo=True)
   - 같은 이벤트의 추가 사진들 → 별도의 단순기록 레코드 (record_text=빈 값, image_url=사진, has_photo=True)
4. 챗봇: "N개의 이벤트로 정리해서 저장했어요!" 응답

### 텍스트 처리 규칙

- 감정분류 모드 다중 이벤트: 이벤트마다 새 텍스트를 요청한다 (이벤트별 감정이 다를 수 있음)
- 단순기록 모드 다중 이벤트: 사용자 텍스트 1개를 모든 이벤트의 주 사진 레코드에 공통으로 적용한다

## 핵심 컴포넌트

### `_parse_photo_exif(url: str) → datetime | None`

- `requests.get(url, stream=True, timeout=5)`로 사진 다운로드
- `piexif`로 `Exif.DateTimeOriginal` (또는 `Image.DateTime`) 파싱
- 형식: `"YYYY:MM:DD HH:MM:SS"` → `datetime` 객체 변환 (KST 가정, 시간대 정보 없음)
- 다운로드 실패, EXIF 없음, 파싱 실패 시 `None` 반환

### `_group_photos_by_event(photos: list[dict]) → list[dict]`

- 입력: `[{"url": str, "exif_time": datetime | None, "received_time": datetime}, ...]`
- 정렬 기준: `exif_time` 우선, 없으면 `received_time` 사용
- 10분 이내(`timedelta(minutes=10)`) 차이 사진은 같은 이벤트
- 출력: `[{"start_time": datetime, "end_time": datetime, "photo_urls": list[str]}, ...]`

### 상태 변수

```python
# 단순기록 모드 사진 누적 버퍼
pending_simple_photo_buffer: dict[str, list[dict]] = {}
# 예: {user_id: [{"url": "...", "received_time": datetime}]}

# 이벤트 그룹화 후 순차 처리 상태
pending_event_groups: dict[str, dict] = {}
# 예: {user_id: {
#     "mode": "emotion" | "simple",
#     "events": [{"start_time": datetime, "photo_urls": [...]}, ...],
#     "current_index": int,
#     "shared_text": str | None,  # 단순기록 모드용
# }}
```

### 진입 조건 체크

webhook에서 텍스트 수신 후, 사진 누적이 2장 이상이면 EXIF 파싱 + 그룹화 진행. 그룹이 1개면 기존 흐름 그대로, 2개 이상이면 `pending_event_groups`에 저장 후 순차 처리 진입.

## 데이터 흐름

```
사용자 → 사진 N장 → pending_photo.urls (감정분류) 또는 pending_simple_photo_buffer (단순기록)
사용자 → 텍스트 → trigger:
  1. _parse_photo_exif(각 사진 URL) 병렬 처리 (asyncio.gather 또는 ThreadPoolExecutor)
  2. _group_photos_by_event 호출
  3. 그룹 1개 → 기존 흐름
  4. 그룹 2+개 → pending_event_groups 저장 → 첫 이벤트 처리 시작
사용자 → 이벤트별 답변 → classify_emotion (감정분류) 또는 단순 저장 (단순기록) → 다음 이벤트로 이동
```

## 에러/엣지 케이스

| 케이스 | 처리 |
|------|----|
| 사진 다운로드 실패 (네트워크, 만료된 URL) | 수신 시간(`received_time`)으로 대체 |
| EXIF에 `DateTimeOriginal` 없음 | 수신 시간 사용 |
| 모든 사진 EXIF 없음 | 1개 이벤트로 처리 (모든 사진이 같은 그룹) |
| 사진 1장만 | 그룹화 스킵, 기존 흐름 그대로 |
| 사용자가 그룹화 흐름 중 다른 명령어 입력 (예: "모드", "내 원석") | `pending_event_groups` 클리어 후 일반 흐름 진입 |
| 단순기록 모드에서 텍스트 없이 사진만 (PHOTO_TIMEOUT 10분 도래) | 자동으로 모든 사진을 텍스트 없이 단일 이벤트로 저장 (현재 `_safe_pending_photo` 타임아웃 로직 활용) |
| 다운로드 시간이 길어 Kakao 타임아웃 위험 | useCallback 패턴 사용 (감정분류와 동일) |

## 응답 메시지 예시

### 감정분류 모드 - 다중 이벤트 진입
```
오늘 2개의 이벤트로 나뉘었어요. ✨

이벤트 1 (오전 11:00 · 사진 3장)
이때 어떤 느낌이었나요?
```

### 감정분류 모드 - 이벤트 진행 중
```
이벤트 2 (오후 6:00 · 사진 2장)
이때는요?
```

### 감정분류 모드 - 모든 이벤트 완료
```
오늘 2개의 이벤트가 모두 저장됐어요! ✨
오늘 5번째 원석이에요! 🪨
```

### 단순기록 모드 - 누적 안내
```
사진 3장 모였어요! 📷
텍스트나 다른 사진을 더 보내주세요.
```

### 단순기록 모드 - 저장 완료
```
2개의 이벤트로 정리해서 저장했어요!
- 오전 11:00 (3장)
- 오후 6:00 (2장)
```

## 검증 시나리오

### 시나리오 1: 사진 1장 (변경 없음)
1. 감정분류 모드에서 사진 1장 → 기존 안내 메시지
2. 텍스트 입력 → 기존 감정 분류 흐름 진행

### 시나리오 2: 사진 다중장 같은 이벤트
1. 감정분류 모드에서 사진 3장 연속 전송 (모두 11:00 ~ 11:05 사이 촬영)
2. 텍스트 입력 → EXIF 파싱 → 1개 이벤트로 그룹화
3. 기존 감정 분류 흐름 진행

### 시나리오 3: 감정분류 모드 다중 이벤트
1. 사진 5장 전송 (11:00 ~ 11:08 사이 3장 + 18:00 ~ 18:03 사이 2장)
2. 텍스트 "오늘 일과" 입력
3. 챗봇: "오늘 2개의 이벤트로 나뉘었어요. 이벤트 1 (11:00 · 3장): 이때 어떤 느낌이었나요?"
4. 사용자: "친구랑 카페 갔어"
5. 챗봇: 감정 원석 제안 → "맞아요" → 저장
6. 챗봇: "이벤트 2 (18:00 · 2장): 이때는요?"
7. 사용자: "혼자 저녁 먹었어"
8. 챗봇: 감정 원석 제안 → "맞아요" → 저장
9. 챗봇: "오늘 2개의 이벤트가 모두 저장됐어요! ✨"
10. DB 확인: `chatbot` 테이블에 2개의 원석 레코드 + 추가 사진 단순기록 레코드 확인

### 시나리오 4: 단순기록 모드 다중 이벤트
1. 단순기록 모드 진입
2. 사진 5장 전송 (다른 이벤트로 그룹화될 시간 분포)
3. 챗봇: "사진 N장 모였어요!" 누적 응답
4. 텍스트 "오늘 일과" 입력
5. 챗봇: EXIF 그룹화 → 각 이벤트 저장
6. 챗봇: "2개의 이벤트로 정리해서 저장했어요!" 응답
7. DB 확인: 각 이벤트마다 주 사진+텍스트 단순기록 1개, 추가 사진 단순기록 N개

### 시나리오 5: EXIF 없는 사진
1. 스크린샷 3장 연속 전송 (EXIF 없음)
2. 텍스트 입력
3. 모두 수신 시간으로 대체되어 1개 이벤트로 그룹화
4. 기존 흐름 진행

### 시나리오 6: 사용자가 그룹화 흐름 중 이탈
1. 다중 이벤트 진입 후 이벤트 1 진행 중
2. 사용자가 "모드" 입력
3. `pending_event_groups` 클리어 + 모드 메뉴 표시

## 향후 확장 가능성 (현재 범위 외)

- GPS 좌표 기반 그룹화 추가 (같은 시간이라도 다른 장소면 다른 이벤트)
- OpenAI Vision API 통합 (사진 내용 기반 감정 분석)
- 이벤트 라벨링 (사용자가 이벤트에 이름 부여)
